# AI_PROMPTS.md

This project was built with substantial assistance from Claude (Anthropic), used as an agentic coding tool with direct access to a sandboxed dev environment (able to write files, run a Python/Node toolchain, start servers, and hit them with real HTTP requests). This document is an honest record of that process, per the assessment's AI-usage disclosure requirement.

## Tool used
Claude (Anthropic), in a chat interface with code-execution/file-creation tools.

## High-level prompt sequence

1. **Initial request** — pasted the assessment brief (objective, key requirements list, recommended stack, submission requirements) and asked for the complete project to be built.
2. **Scope clarification** — Claude flagged that the "attached" detailed assessment document referenced in the brief hadn't actually come through (only the summary text had), and proceeded on the summary alone, explicitly documenting every assumption instead of blocking on the missing spec.
3. **"give me complete code" / "continue"** — Claude scaffolded the full repo: FastAPI backend (models, schemas, 7 routers, seed script), Next.js frontend (dashboard, employees, projects, seats, new joiners, AI assistant pages), Dockerfile, `.env.example` files, and this documentation.

## What was generated vs. manually fixed

**Generated by Claude (near-final, see fixes below):**
- All SQLAlchemy models (`backend/app/models.py`) — 8 tables covering departments, employees, projects, project_assignments, floors, seats, seat_allocations, new_joiner_requests.
- All Pydantic schemas (`backend/app/schemas.py`).
- All 7 FastAPI routers (employees, projects, seats, new_joiners, dashboard, search, assistant), including business-rule enforcement (double-booking prevention, one-seat-per-employee, allocation % cap, offboarding cascade).
- Seed data generator (`backend/app/seed.py`) using Faker, producing ~5,000 employees / ~150 projects / ~5,200 seats / assignments / allocations / pending new-joiner requests in a few seconds.
- Full Next.js frontend: typed API client, dashboard with Recharts visualizations, employee/project/seat/new-joiner CRUD pages, AI assistant chat UI, sidebar/topbar/stat-card components, Tailwind styling.
- The rule-based + Claude-fallback design for the AI Assistant endpoint (see README's "AI Assistant Design" section for the reasoning — deliberately never lets an LLM execute raw SQL).

**Found and fixed during validation** (Claude ran the actual code rather than just generating it and stopping):
1. **Case-sensitive employee-code lookup bug** in `assistant.py`: the regex parser uppercased/lowercased captured names before an exact-match (`==`) query against `employee_code`, so `"where does ETH1001 sit"` failed to resolve even though the employee existed. *Prompt/action:* Claude wrote and ran a live query — `curl -X POST /api/assistant/query -d '{"query":"where does ETH1001 sit"}'` — got `"I couldn't find that employee"`, traced it to the `==` comparison, and changed it to `.ilike()`.
2. Same bug, same root cause, in the `who_sits_at_seat` intent (`seat_code == "FLOOR1-A-001"` vs. stored `"Floor1-A-001"`). Fixed the same way.
3. **Missing regex trigger for department headcount**: `headcount_of_department` was fully implemented in `_execute_intent` but had no regex pattern routing to it, so it always fell through to "unknown" without an Anthropic API key configured. Added `r"how many (?:people|employees|members).*department[:\s]+([\w\s]+)"`.
4. **Stray directory from a broken shell brace-expansion** (`mkdir app/{employees/[id],projects,...}` had apparently been run in a shell that didn't expand braces, leaving a literal folder named `{employees/[id],projects,seats,new-joiners,assistant}`). Found via `find`, removed — cosmetic/repo-hygiene only, didn't affect the running app.

## Post-build change: Anthropic → Groq for the AI Assistant fallback

**Prompt:** "i did not want to use anthropic api i want to use groq api"

The assistant's optional LLM fallback (used only when the rule-based parser doesn't match a query) originally called the Anthropic API. Swapped to Groq's OpenAI-compatible chat completions API (`llama-3.3-70b-versatile`, JSON mode) instead — matches the free-tier/Groq stack already used elsewhere in this developer's other projects. Changes made:
- `requirements.txt`: `anthropic` → `groq`
- `assistant.py`: `_claude_parse()` → `_groq_parse()`, `ANTHROPIC_API_KEY` → `GROQ_API_KEY` (+ `GROQ_MODEL`, defaulting to `llama-3.3-70b-versatile`)
- `.env.example` and README updated to match

*Validation:* reinstalled `requirements.txt` into a fresh venv, reseeded, and ran the server live — confirmed the rule-based intents still work identically and that unmatched queries fail gracefully with no key set. The actual Groq API call path could **not** be live-tested from this sandbox (`api.groq.com` isn't on the environment's outbound network allowlist), so test this end-to-end with a real `GROQ_API_KEY` before relying on the LLM-fallback path specifically.

## Validation methods used

- **Backend**: created a Python venv, installed `requirements.txt`, ran the seed script at both small scale (300 employees, for fast iteration) and full scale (5,000 employees, 3 seconds), then started a live `uvicorn` server and hit it with `curl` for:
  - Every GET endpoint (dashboard summary/utilization/headcount, employees list+detail, seats list+available, projects list, search, new-joiners list).
  - Every write path: seat allocate → double-allocate (expect 400) → release → double-release (expect 404); new-joiner creation with auto-allocation; project assignment with the >100% rule triggered for real (not just a single out-of-range field, which FastAPI/Pydantic already 422s automatically); duplicate-email employee creation (expect 400); employee offboarding, then re-fetched the employee to confirm `employment_status` flipped to `exited` and their seat was actually released.
  - All AI assistant intents by phrasing a natural-language query for each one and checking the returned `interpreted_as` field matched the intended intent.
  - Performance at 5,000-employee scale: dashboard summary ~40ms, employee search ~14ms, paginated seat listing ~47ms — all well within acceptable range for an internal tool.
- **Frontend**: ran `npx tsc --noEmit` (zero errors) and a full `next build` (zero errors, all 9 routes compiled/prerendered) — the Google Fonts import was temporarily swapped for a build-only test since this sandbox has no outbound access to `fonts.googleapis.com`; reverted immediately after confirming the rest of the build was clean. Grepped every page for its API calls and cross-checked each path/method against the live-tested backend routes to confirm the two sides actually agree with each other.

## Known gaps not covered by this AI-assisted pass

- No automated test suite (pytest/Jest) was written — validation was manual/live-request-based as described above, not regression-protected.
- No auth layer, so role-based access (Employee/HR/Admin/Project team from the brief) is not implemented.
- Live deployment (Railway/Render/Vercel URLs) was not carried out from within this environment — no ability to create external accounts or push to a real GitHub repo from here. See README for exact deployment steps to run manually.
- Screenshots were not produced: no internet access to download a headless browser in this sandbox (`npx playwright install chromium` was attempted and silently no-opped due to network restrictions).
